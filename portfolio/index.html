개발자의 .env 시크릿을 기기에 AES-256-GCM으로 암호화 저장하고, Claude(Desktop·Code)에 MCP로 노출하되 값이 나갈 때마다 사용자 승인을 강제하는 로컬 우선 데스크톱 앱. Paddle 결제 → 서명 라이선스 발급 → 오프라인 검증까지 엔드투엔드로 구현.

Problem

AI 시대의 새로운 시크릿 문제

AI 코딩 도구가 대중화되며 "내 API 키·토큰을 AI에게 안전하게 넘기는 법"이 새 과제가 됐다. 기존 매니저는 시크릿을 클라우드 계정에 업로드해야 하고, AI에게 값을 넘기는 순간 사용자 통제권이 사라진다.

Solution

"내 기기에만, 넘길 땐 내가 허락"

시크릿은 오직 기기에 암호화되어 있고, AI가 값을 읽으려면 매번 네이티브 승인 다이얼로그를 통과해야 한다. 거부·타임아웃·다이얼로그 실패는 모두 값 미방출(fail-closed). 이 원칙을 코드로 강제했다.

결제 시스템 — Paddle 엔드투엔드

payments/webhook · Rust
■ License Issuance — Paddle → Ed25519 → offline verify▢✕

Merchant of Record인 Paddle Billing을 결제 게이트웨이로 쓰고, 결제 이벤트를 받아 자체 서명 라이선스를 발급하는 독립 Rust 웹훅을 직접 구현했다. 구독이 아닌 1회 결제($59) 평생 라이선스.

  • 1

    결제 완료 — transaction.completed 수신, Paddle-Signature(HMAC-SHA256) 검증 및 오래된 타임스탬프 거부.

  • 2

    라이선스 발급 — price id → 플랜 매핑, 짧은 코드 생성 + Ed25519 서명 인증서 발급(코어 로직 재사용).

  • 3

    저장 & 발송 — Supabase에 저장(transaction_id 유니크 = 멱등성), Resend로 코드·딥링크 이메일 발송.

  • 4

    오프라인 활성화 — 앱이 코드+이메일로 활성화, 인증서를 기기에서 오프라인 검증 후 Pro 해제. 이후 에어갭 재검증.

  • 5

    환불·차지백 — adjustment.created로 해당 라이선스 revoked 처리, 신규 활성화 차단.

멱등성

transaction_id DB 유니크 제약으로 Paddle 재시도·중복 전송이 라이선스를 이중 발급하지 않는다.

결제 손실 방지

일시적 실패는 HTTP 503으로 재시도 유도. 행 삽입·이메일 발송이 끝난 뒤에만 200 반환.

키 관리

서명 개인키는 서버 시크릿에만, 앱엔 공개키만 내장. 릴리스 전 checkkey로 수락 여부 검증.

정책 불변식

무료 티어 상한 같은 규칙은 단일 공유 술어(can_write_variable)로 구현해 GUI 경로와 MCP 쓰기 경로가 절대 어긋나지 않게 했다.

보안 모델 — 정직하게

과장하지 않고, 한계까지 문서화
저장 시 암호화

AES-256-GCM 봉투(nonce·ciphertext)만 디스크에. 평문은 파일에 닿지 않음.

키 파생

기본 기기 바인딩(SHA-256), 옵션으로 Argon2id 마스터패스워드 볼트(메모리-하드).

fail-closed 승인

거부·타임아웃·다이얼로그 실패 = 전부 값 미방출. fail-open 경로 없음.

범위 제한 읽기

read_env_variables는 변수명을 정확히 지정해야 함(와일드카드 거부), 부분 승인 가능.

위조 방지

앱은 공개키만 내장 → 검증은 가능하되 라이선스 발급(minting)은 불가능.

한계 명시

기기 바인딩은 파일 복사는 막지만 로컬 코드 실행 공격은 못 막음을 문서에 명시.
